Root Cause

TlsPskUtils.java — the vulnerable method
// BEFORE (vulnerable)
protected static byte[] getAppDataBytesAndRelease(ByteBuf byteBufMsg) {
    byte[] appDataBytes = byteBufMsg.hasArray()
        ? byteBufMsg.array()           // ← full backing array (pool chunk capacity)
        : TlsPskUtils.readDirect(byteBufMsg);
    ReferenceCountUtil.safeRelease(byteBufMsg);  // ← freed before use
    return appDataBytes;               // ← returns freed array
}

// TlsPskHandler.write then calls:
tlsPskServerProtocol.writeApplicationData(appDataBytes, 0, appDataBytes.length);
//                                                       ↑           ↑
//                                               offset=0  len=capacity (NOT readableBytes)

Use-After-Free

safeRelease is called before writeApplicationData consumes the array. The Netty pool can reallocate that memory to another channel mid-encryption, causing a data race and potential cross-channel data corruption.

Buffer Over-Read (Heartbleed-style)

appDataBytes.length is the pool chunk capacity (e.g. 2048, 4096, or 16384 bytes), not readableBytes(). The extra bytes are stale heap data — HTTP headers, tokens, and cookies from earlier requests on that pooled buffer — encrypted and sent to the attacker verbatim.
